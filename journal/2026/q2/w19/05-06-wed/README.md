- Z3WIRE-LE-BE-BYTE-API: explicit-endianness API at call site (old API was implicit-LE, commented only). Right-aligned partial-byte convention (Option A, matches Python `int.to_bytes`, Java `BigInteger`, GMP, Verilog; ASN.1 BIT STRING uses left-aligned but is self-describing — not counter-precedent). BE = byte-reverse of LE → impl reverses input/output and delegates to LE path. Variable-length input span: `< kNumBytes` zero-pad on high-magnitude end no truncation; `== kNumBytes` truncate iff partial-byte garbage in unused high bits; `> kNumBytes` truncate iff any extra high-magnitude bytes non-zero or partial-byte garbage (s01)
- Z3WIRE-NAMING-RULE-FINAL: 3 buckets — (1) free functions: snake_case (matches Z3 `z3::shl`/`ashr`/`concat`/`ite`/`zext`/`mk_and` + std::, also SystemC `sc_start`, Bitwuzla `mk_bv_value`); (2) class methods: Google C++ Style PascalCase if non-accessor, snake_case if accessor (`value()`, `expr()` are accessors); (3) type traits: std-style snake_case + `_v`. Justification: "calls like `z3::shl` and `z3w::shl` read consistently." User dropped overclaim about "other SMT/HW libraries in the ecosystem" after audit found camelCase counterexamples: Verilator (`assertOn`/`gotFinish`), MLIR/CIRCT (`emitError`/`emitWarning`), CVC5 (`mkBitVector` — methods, not free fns) (s01, s02)
- Z3WIRE-MEMBER-VS-FREE-FN-FOR-TO-BYTES: `ToLeBytes`/`ToBeBytes` as members not free fns. Reasons: (a) needs internal storage access — member avoids leaking via friend or accessor; (b) symmetry with `FromLeBytes`/`FromBeBytes` static factories; (c) discoverability on type's interface. The free-fn shape of `to_symbolic`/`to_concrete` is forced by `bit_vec.h` not depending on Z3 — that constraint doesn't apply to byte serialization (s01)
