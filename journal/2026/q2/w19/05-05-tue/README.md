## Agent index

- Z3WIRE-CONSTRUCTION-API: `Literal<V>()` (NTTP, static_assert) / `From(v)` (abort on out-of-range) / `TryFrom(v)` (recoverable). Bare `From` over `FromValue`/`FromExpr` — Rust idiom + overloading-on-param-type disambiguates concrete vs symbolic. `Literal<V>()` kept as NTTP (consteval ctor would lose static_assert clarity — function params aren't constant expressions even inside consteval). No Abseil dep (heavyweight for one feature). Bit-truncation, not clamping (matches HDL conventions, Z3 BV `extract`, z3wire arithmetic) — future `saturating_cast` tracked in roadmap (s01)
- Z3WIRE-IMPLEMENTER-DEVIATION-CORRECT: in `to_concrete`, plan said `From()` but implementer used `TryFrom(...).value`. Reason: Z3's `get_numeral_uint64()` returns bit pattern as unsigned; for signed types, valid negative values produce uint64 > max_signed → `From()` would spuriously abort. Original code `std::get<0>(FromValue(...))` silently discarded truncation flag. Lesson: trust subagent judgment when reasoning is sound (s01)
- BAZEL-LOCKFILE-MUST-COMMIT: if `MODULE.bazel.lock` gitignored, `hashFiles('MODULE.bazel.lock')` returns empty-set hash on every CI run → `actions/cache@v4` key never invalidates → cache silently grows stale on dep updates. Bazel 8+ official guidance is to commit Bzlmod lockfiles anyway. Z3Wire was on 8.6.0 (s02)
- ACTIONS-CACHE-V4-MECHANICS: step does both restore-at-step-time + implicit save as post-step. Save only fires if (a) job succeeded AND (b) exact `key` didn't already exist at job start. No save on failure → broken first run can't poison cache. GitHub repo cache budget: 10 GB LRU. Step name "Restore Bazel disk cache" misleading (also saves) but matches existing `cmake.yml` "Restore ccache" convention (s02)
