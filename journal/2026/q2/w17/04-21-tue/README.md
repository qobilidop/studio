## Agent index

- SYMEX-SURVEY: Baldoni et al. arXiv:1610.00502 confirmed gold standard. Newer surveys (2024-2025) focus on domain applications (smart contracts, IoT firmware) not engine architecture. Emerging: LLM-symbolic hybrid (AAAI 2026), CPS-based parallel compilation (ICSE 2023) (s00)
- ANYDSL-IMPALA: AnyDSL framework — Impala frontend → Thorin CPS IR with online partial evaluator → LLVM `.ll`. `@` annotations force compile-time specialization. Notable DSLs: Rodent (ray tracer, single codebase → CPU SIMD + GPU CUDA, matches Embree/OptiX), HIPAcc image processing (25-50% faster than OpenCV), genome sequence alignment (matches SeqAn), AnyDSL-MD molecular dynamics. `artic` is newer Rust-like syntax for Thorin (s00)
- MIMIR: AnyDSL's next-gen IR, derived from Thorin but with dependent types (Calculus of Constructions PTS), supports both CPS and direct style, has own surface language `Mim`, plugin-based axiom extension. Better fit for P4 than Thorin because dependent types natively encode bit-widths (s00)
- LLVM-BITVEC: LLVM natively supports `iN` for any N up to ~i8388607 (compile-time slow past few thousand). Register-level operations respect width (no manual masking needed). Memory storage NOT packed — `store i9` likely pads to 16-bit per DataLayout. For P4 packed headers must emit manual shift/mask/trunc. `varbit<W>` requires struct of fixed-max-int + length tracker (s00)
- MLIR-CIRCT-PACKED: CIRCT solves LLVM's packing problem. `comb` dialect = strict hardware-semantic arithmetic on signless integers (no implicit promotion). `hw.struct`/`hw.array` strictly packed — never padded. Path: P4 → custom `p4.arith`/`p4.header` out-of-tree dialect → lower to CIRCT `comb`/`hw` → LLVM `.ll` or SystemVerilog. Modern network compilers migrating LLVM→MLIR (s00)
- P4-MLIR-STRATEGY: contributing `p4arith` upstream to CIRCT discouraged — domain dialects bloat. Standard pattern: out-of-tree compiler depending on LLVM/MLIR + CIRCT. Existing p4mlir-incubator under p4lang org is exact precedent. User considering this direction (s00)
- IMPALA-FOR-P4: shallow embedding pitch — write P4 abstractions as Impala higher-order functions w/ `@`, get `extern "C"` + LLVM `.ll` + cross-compile to GPU "for free". Concrete example: parser states as continuation-passing functions, Thorin flattens to single basic block. Pivots paradigm from "build P4 compiler" → "build P4 library inside DSL-optimizing compiler" (s00)
- PYTHON-FFI: `ctypes` (stdlib, manual type mapping), `cffi` (parses raw C headers, perfect with `--emit-c-interface`, zero-copy via `ffi.from_buffer`), Cython (`cdef extern from`, max perf, compiles to C-API). cffi is best fit for P4Py auto-generated headers (s00)
- TVM-ARCH: Relay/Relax (functional graph IR, op fusion, dynamic shapes in Relax) → TIR (imperative loop IR for scheduling). Halide-inspired algorithm/schedule separation. AutoTVM/MetaSchedule auto-search. Frontends: ONNX/TF/PyTorch FX/JAX. C++ AST exposed to Python via FFI (s00)
- ONNX-DEEPDIVE: protobuf schema, Opsets for portability, FunctionProto local functions (IR v8+), Loop operator embeds body subgraph (`max_trip_count`+`condition_var`+loop-carried deps+`scan_outputs`), custom ops via domain namespace + C++ kernel registration. Strictly stateless inference — externalize state through input/output tensors. Training Extension allows mutation but inference engines reject it (s00)
- ONNX-RUNTIME: Microsoft's ORT — graph partitioning across Execution Providers (CUDA/TensorRT/CoreML/OpenVINO/QNN), CPU fallback. Custom ops via FFI to C++ shared lib. Aggressive parallelism breaks if custom op secretly mutates global state (race conditions, DCE). MLIR/CIRCT structurally superior for stateful network IR (s00)
- NETWORKING-DSLS: P4 (Match-Action pipeline), NetKAT (Kleene Algebra w/ Tests, mathematical verification), Lucid (event-driven imperative → compiles to P4 or C), NV (control-plane SMT verification), NPL (Broadcom, hardware-bound), eBPF/XDP (restricted C in kernel). All structurally divergent (s00)
- NET-ONNX-FEASIBILITY: hypothetical universal protobuf for networking DSLs. Hard because no shared math foundation (P4=pipeline, Lucid=events, NetKAT=algebra, eBPF=ISA). P4Info exists but only describes table interfaces, not data plane. MLIR currently winning the universal-network-IR war. Could decouple frontends from backends if scoped to data plane only (s00)
- ABBREV-SEMANTIC: `sema` (Clang/LLVM gold standard, recommended for compiler passes), `sem` (generic), avoid `smt` (collides w/ SMT solvers) and `sm` (state machine/shared memory) (s00)
