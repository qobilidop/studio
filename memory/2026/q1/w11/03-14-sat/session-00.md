# P4kt

[![CI](https://github.com/qobilidop/p4kt/actions/workflows/ci.yml/badge.svg)](https://github.com/qobilidop/p4kt/actions/workflows/ci.yml)
[![Docs](https://github.com/qobilidop/p4kt/actions/workflows/docs.yml/badge.svg)](https://qobilidop.github.io/p4kt/)

P4kt (pronounced "packet") is an embedded domain-specific language (eDSL) for [P4](https://p4.org) in Kotlin.

## Why eDSL for P4?

P4 is great at describing packet processing, but it has no way to programmatically generate or compose designs beyond what the language's type system allows. P4 does support the C preprocessor, but preprocessor macros are fragile, untyped, and error-prone.

An eDSL gives you the full power of a general-purpose language while staying in the P4 domain. You write P4 concepts - headers, parsers, tables - but with the host language's abstraction, composition, and tooling.

## How it works

P4kt is a code generation eDSL. You write Kotlin that _describes_ a P4 program, then render it to P4 source text. The output feeds into the existing P4 toolchain - compilers, simulators, and hardware targets all work unchanged. Generated P4 can be mixed with hand-written P4 or output from other tools, and the output is always inspectable.

## Why Kotlin?

P4 programs are hierarchical and declarative - packages contain controls, controls contain tables, tables contain actions. The host language for a P4 eDSL needs strong builder-pattern support more than deep embedding features like monads or type-level programming.

Here's how the alternatives compare:

- **Haskell** - the gold standard for deeply embedded DSLs, with type classes, monads, and higher-kinded types. [Clash](https://clash-lang.org/) uses Haskell's functional abstractions to describe hardware and compiles to VHDL/Verilog. But these features are more power than a P4 code generation DSL needs, and the high barrier to entry is hard to justify when simpler tools suffice.
- **Scala** - the strongest JVM language for eDSLs, with implicit conversions, macros, and higher-kinded types. [Chisel](https://www.chisel-lang.org/) and [SpinalHDL](https://github.com/SpinalHDL/SpinalHDL), both hardware description eDSLs, demonstrate Scala's strength in a neighboring domain. But Scala has a steeper learning curve and smaller community than Kotlin.
- **OCaml** - excellent for compiler and DSL work, with a powerful module system and algebraic data types. OCaml has real presence in the P4 ecosystem - [Petr4](https://github.com/verified-network-toolchain/petr4) provides formal semantics and a definitional interpreter for P4, and [Hardcaml](https://github.com/janestreet/hardcaml) is a hardware eDSL. But OCaml's strength is in analysis and verification; its lack of type classes and smaller ecosystem make it less suited for ergonomic builder-style eDSLs.
- **Rust** - powerful type system, but its eDSL approach relies on procedural macros operating on token streams - closer to code generation than true embedding. Builder-pattern DSLs are verbose compared to Kotlin's lambda receivers.
- **C++** - the dominant language in networking and the language of the P4 reference compiler [p4c](https://github.com/p4lang/p4c). Template metaprogramming enables powerful eDSLs like [Boost.Spirit](https://www.boost.org/doc/libs/release/libs/spirit/), but they are notoriously hard to write, debug, and read.
- **Python** - a strong alternative with different tradeoffs. Python eDSLs like [Amaranth HDL](https://amaranth-lang.org/) perform rich domain-specific checking at elaboration time, catching width mismatches and invalid connections with clear messages. Python offers a lower barrier to entry, a massive ecosystem, and no compile step. The tradeoff is that structural errors (typos in names, wrong types passed to builders) surface when you run the script rather than in the IDE as you type.

Kotlin hits the sweet spot. Its lambda receivers, extension functions, property delegates, and operator overloading are purpose-built for builder DSLs - exactly the pattern P4's hierarchical structure calls for. It's more accessible than Haskell or Scala while being far more expressive for eDSLs than C++ or Rust. Compared to Python, Kotlin catches structural errors at compile time with IDE feedback, though domain-specific checks like bit-width validation happen at code generation time in both approaches. Kotlin is also the language behind [Gradle's Kotlin DSL](https://docs.gradle.org/current/userguide/kotlin_dsl.html) and [Jetpack Compose](https://developer.android.com/compose), two of the most widely-used eDSLs in industry.

If hitting `.` doesn't immediately bring up exactly the right P4 constructs, or if the auto-complete is flooded with irrelevant methods, the entire value proposition of using Kotlin collapses. You might as well just write raw P4 and deal with the C preprocessor. The IDE experience *is* the product when it comes to eDSLs.

Getting this right in Kotlin usually comes down to tightly controlling the scope of your lambda receivers so the IDE doesn't get confused. Here are a few common traps and patterns to look out for with Claude Code as you iterate on the design:

### 1. The Scope Leakage Problem (`@DslMarker`)
If you have nested builders (e.g., a `control` block containing a `table` block), Kotlin's default behavior allows the inner block to access all the methods of the outer block. This completely trashes auto-complete because typing inside a `table` will suggest control-level functions. 
* **The Fix:** You absolutely need to use `@DslMarker`. By annotating your builder interfaces with a custom DSL marker, you force the IDE to restrict auto-complete exclusively to the closest receiver scope. 

### 2. The "Stringly Typed" Trap
P4 relies heavily on referencing identifiers (e.g., a table referencing an action, or a parser extracting a specific header). If your DSL relies on passing strings to make these connections (e.g., `actions("my_routing_action")`), the IDE can't help the user.
* **The Fix:** Force the use of strongly typed object references. If an action is defined as a Kotlin `val`, passing that `val` directly into the table's action list allows the IDE to infer the type and provide auto-complete for only valid `Action` objects. 

### 3. Interface Segregation over Massive Builders
If your core `P4ProgramBuilder` interface has 50 different methods for every single P4 concept, auto-complete becomes a nightmare to visually parse. 
* **The Fix:** Break the builders down. A `ParserBuilder` should only know about `state`, `extract`, and `transition`. Keeping the receiver interfaces incredibly narrow ensures the IDE only suggests what is legally allowed at that exact node in the AST.

It is a tricky balancing act between making the AST easy to build under the hood and making the front-end syntax ergonomic for the user.

# gemini response

**Timestamp: 9:49 PM PDT**

You have hit on one of the most fundamental architectural crossroads in eDSL design: the boundary between the host language's type system and the target language's AST. 

You are absolutely right that using native Kotlin classes is not the traditional builder pattern. But here is the critical takeaway: **it is exactly the right approach, and it is a feature, not a bug.**

Here is why you and Claude Code actually made the optimal architectural choice:

### The Limits of the Builder Pattern
The builder pattern (using lambda receivers like `control { ... }` or `apply { ... }`) is undefeated for defining *hierarchical structure* and *control flow*. 

However, it completely falls apart for *data type definitions* because builders generate AST nodes at runtime. If you define a struct using a builder block, the Kotlin compiler only sees a generic `StructNode` object. When you later try to type `myHeader.ethernet.srcAddr`, the IDE throws an error because the Kotlin compiler has no idea what `ethernet` is—that information is trapped inside your runtime AST.

### The "Native Class" Paradigm
By using native Kotlin classes to represent P4 structs, you are forcing the host language (Kotlin) to do the heavy lifting for the target language (P4). 
* **Static Analysis:** Because it is a real Kotlin class, the IDE's static analyzer can immediately provide auto-complete for field access. 
* **Type Safety:** You get compile-time guarantees that a user isn't trying to access an IPv6 field on an IPv4 header. 
* **The eDSL Precedent:** This is exactly how the most mature Kotlin eDSLs handle data schemas. For example, JetBrains Exposed (the standard Kotlin SQL DSL) requires you to define database tables as native Kotlin `object` or `class` types specifically so you can get native auto-complete for SQL queries like `Users.name.eq("Bili")`. 

### Bridging the Gap: Property Delegation
To keep this native class approach feeling like a slick DSL rather than clunky boilerplate, the idiomatic Kotlin move is to use **Property Delegation** (`by`). This allows you to define the native class for the IDE, while under the hood, it automatically wires the fields into your P4 AST.

It usually looks something like this:
```kotlin
class EthernetHeader : P4Struct() {
    val dstAddr by bit(48)
    val srcAddr by bit(48)
    val etherType by bit(16)
}
```
In this setup, `bit(48)` is a delegate that simultaneously registers the field in the underlying AST model *and* exposes it as a strongly typed Kotlin property for perfect IDE auto-complete.
